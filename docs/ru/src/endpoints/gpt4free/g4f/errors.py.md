# Модуль errors

## Обзор

Модуль `errors` содержит определения пользовательских исключений, используемых в проекте `hypotez`. Эти исключения позволяют обрабатывать специфические ошибки, которые могут возникать при взаимодействии с различными провайдерами, моделями и API.

## Подробнее

Модуль определяет ряд исключений, каждое из которых предназначено для конкретной ситуации. Использование этих исключений позволяет более точно обрабатывать ошибки и предоставляет информацию о причинах их возникновения.

## Классы

### `ProviderNotFoundError`

**Описание**: Исключение, которое вызывается, когда провайдер не найден.

**Принцип работы**:
Класс `ProviderNotFoundError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что запрошенный провайдер не был найден в системе.

### `ProviderNotWorkingError`

**Описание**: Исключение, которое вызывается, когда провайдер не работает.

**Принцип работы**:
Класс `ProviderNotWorkingError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что провайдер, который должен быть активен, не функционирует должным образом.

### `StreamNotSupportedError`

**Описание**: Исключение, которое вызывается, когда потоковая передача не поддерживается.

**Принцип работы**:
Класс `StreamNotSupportedError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что запрошенный метод потоковой передачи не поддерживается.

### `ModelNotFoundError`

**Описание**: Исключение, которое вызывается, когда модель не найдена.

**Принцип работы**:
Класс `ModelNotFoundError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что запрошенная модель не была найдена в системе.

### `ModelNotAllowedError`

**Описание**: Исключение, которое вызывается, когда модель не разрешена.

**Принцип работы**:
Класс `ModelNotAllowedError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что использование запрошенной модели не разрешено в данном контексте.

### `RetryProviderError`

**Описание**: Исключение, которое вызывается, когда необходимо повторить попытку с провайдером.

**Принцип работы**:
Класс `RetryProviderError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что необходимо повторить попытку выполнения операции с текущим провайдером.

### `RetryNoProviderError`

**Описание**: Исключение, которое вызывается, когда необходимо повторить попытку без провайдера.

**Принцип работы**:
Класс `RetryNoProviderError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что необходимо повторить попытку выполнения операции без указания конкретного провайдера.

### `VersionNotFoundError`

**Описание**: Исключение, которое вызывается, когда версия не найдена.

**Принцип работы**:
Класс `VersionNotFoundError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что запрошенная версия чего-либо не была найдена в системе.

### `ModelNotSupportedError`

**Описание**: Исключение, которое вызывается, когда модель не поддерживается.

**Принцип работы**:
Класс `ModelNotSupportedError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что запрошенная модель не поддерживается в данном контексте.

### `MissingRequirementsError`

**Описание**: Исключение, которое вызывается, когда отсутствуют необходимые зависимости.

**Принцип работы**:
Класс `MissingRequirementsError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что для выполнения операции отсутствуют необходимые зависимости или библиотеки.

### `NestAsyncioError`

**Принцип работы**:
Класс `NestAsyncioError` является подклассом класса `MissingRequirementsError`. Он используется для сигнализации о проблемах, связанных с асинхронностью, например, когда требуется использовать `nest_asyncio`.

**Принцип работы**:
Класс `MissingAuthError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что для выполнения операции требуется аутентификация, но она не была предоставлена.

### `PaymentRequiredError`

**Описание**: Исключение, которое вызывается, когда отсутствует медиа-ответ.

**Принцип работы**:
Класс `NoMediaResponseError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что при запросе медиа-контента не был получен ответ.

### `ResponseError`

**Описание**: Исключение, которое вызывается при ошибке ответа.

**Принцип работы**:
Класс `ResponseError` является подклассом стандартного класса `Exception`. Он используется для сигнализации об общей ошибке, связанной с ответом от API или другого сервиса.

### `ResponseStatusError`

**Описание**: Исключение, которое вызывается при ошибке статуса ответа.

**Принцип работы**:
Класс `ResponseStatusError` является подклассом класса `ResponseError`. Он используется для сигнализации об ошибке, связанной со статусом ответа, например, когда статус код указывает на ошибку.

### `RateLimitError`

**Описание**: Исключение, которое вызывается при превышении лимита запросов.

**Принцип работы**:
Класс `RateLimitError` является подклассом класса `ResponseStatusError`. Он используется для сигнализации о том, что было превышено ограничение на количество запросов к API или сервису.

**Описание**: Исключение, которое вызывается, когда отсутствует валидный HAR-файл.

**Принцип работы**:
Класс `NoValidHarFileError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что предоставленный HAR-файл (HTTP Archive) не является валидным или не может быть обработан.

**Описание**: Исключение, которое вызывается при превышении лимита на количество разговоров при запросах API к AI endpoint.

**Принцип работы**:
Класс `ConversationLimitError` является подклассом стандартного класса `Exception`. Он используется для сигнализации о том, что достигнут лимит на количество разговоров при взаимодействии с AI endpoint.
