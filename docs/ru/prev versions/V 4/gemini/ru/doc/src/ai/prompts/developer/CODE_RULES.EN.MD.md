# Документация для `CODE_RULES.EN.MD`

## Обзор

Этот файл содержит инструкции для разработчиков по созданию, анализу и улучшению кода в проекте `hypotez`. Он определяет основные принципы и требования к стилю кодирования, форматированию и документированию кода.

## Подробней

Файл `CODE_RULES.EN.MD` служит руководством для поддержания единообразного и читаемого стиля кодирования во всем проекте `hypotez`. Он содержит правила и рекомендации по форматированию кода, комментированию, обработке ошибок и логированию, а также примеры правильного и неправильного использования определенных конструкций. Следование этим правилам помогает улучшить качество кода, облегчить его понимание и сопровождение, а также снизить вероятность ошибок.

## Разделы

### 1. Основные принципы

#### 1.1 Общие указания

- Необходимо придерживаться четкого стиля кодирования.
- Все изменения должны быть обоснованы и соответствовать требованиям.

#### 1.2 Комментарии

- Используйте `#` для внутренних комментариев.
- Документация для всех функций, методов и классов должна следовать формату **Sphinx**, основанному на **reStructuredText (reST)**.
- Комментарии и документация должны быть четкими, краткими и точными.

#### 1.3 Форматирование кода

- Используйте одинарные кавычки.
- Добавляйте пробелы вокруг операторов.
- Все параметры должны быть типизированы.
- Избегайте использования `Union`.

#### 1.4 Логирование

- Используйте модуль `logger` из `src.logger.logger`.
- Примеры ошибок должны регистрироваться с использованием `logger.error`.

### 2. Основные требования

#### 2.1 Формат Markdown для ответов

- Все ответы должны быть в формате **Markdown**.
- Структура ответа должна включать:
  - **Исходный код**: Блок полученного кода без изменений.
  - **Улучшенный код**: Блок улучшенного кода, правильно отформатированный и с добавленными комментариями.
  - **Изменения**: Подробный список изменений и пояснений.
- Код должен быть заключен в соответствующие теги подсветки синтаксиса (например, `python`, `markdown`, `json`).

#### 2.2 Формат комментариев

- Используйте определенный стиль для комментариев и документации в коде.
- Пример:

```python
    def read_text_file(
        file_path: str | Path,
        as_list: bool = False,
        extensions: Optional[list[str]] = None,
        chunk_size: int = 8192
    ) -> Generator[str, None, None] | str | None:
        """
        Reads the contents of a file (or files from a directory) using a generator for memory efficiency.

        :param file_path: Path to the file or directory.
        :param as_list: If `True`, returns a generator of strings.
        :param extensions: List of file extensions to include when reading a directory.
        :param chunk_size: The size of chunks to read the file in bytes.

        :returns: A string generator, a concatenated string, or `None` in case of error.

        :raises Exception: If an error occurs while reading the file.

        Example:
            >>> from pathlib import Path
            >>> file_path = Path('example.txt')
            >>> content = read_text_file(file_path)
            >>> if content:
            ...    print(f'File content: {content[:100]}...')
            File content: Example text...
        """
        ...
```

- Всегда предоставляйте подробные объяснения в комментариях. Избегайте расплывчатых терминов, таких как *"получить"* или *"делать"*. Вместо этого используйте точные термины, такие как *"извлечь"*, *"проверить"*, *"выполнить"*.
- Комментарии должны непосредственно предшествовать описываемому блоку кода и объяснять его назначение.

#### 2.3 Пробелы вокруг операторов присваивания

- Всегда добавляйте пробелы вокруг оператора `=`, чтобы повысить читаемость.
- Примеры:
  - **Неправильно**: `x=5`
  - **Правильно**: `x = 5`

#### 2.4 Использование `j_loads` или `j_loads_ns`

- Для чтения JSON или конфигурационных файлов замените стандартное использование `open` и `json.load` на `j_loads` или `j_loads_ns`.
- Пример:

```python
# Incorrect:
with open('config.json', 'r', encoding='utf-8') as f:
    data = json.load(f)

# Correct:
data = j_loads('config.json')
```

#### 2.5 Сохранение комментариев

- Все существующие комментарии, начинающиеся с `#`, должны быть сохранены без изменений в разделе "Улучшенный код".
- Если комментарий кажется устаревшим или неясным, не изменяйте его. Вместо этого отметьте его в разделе "Изменения".

#### 2.6 Обработка `...` в коде

- Оставляйте `...` как указатели в коде без изменений.
- Не документируйте строки с `...`.
